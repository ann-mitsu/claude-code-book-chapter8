# プロダクト要求定義書(PRD)

## 1. プロダクト概要

### 1.1 名称

**TaskCLI** - Gitと一体化したタスク管理CLIツール

### 1.2 プロダクトコンセプト

- **ターミナル完結**: 画面切り替えゼロ。コード作成とタスク管理を同じ環境で
- **Git統合**: タスクとブランチが自動連携。コミット、PR、マージの流れとタスクが一体化
- **開発者ファースト**: シンプルで高速。1分で使い始められる軽量設計

### 1.3 プロダクトビジョン

開発者がターミナルから離れることなく、タスク管理と開発作業をシームレスに行えるCLIツールを提供する。Gitとの深い統合により、タスク開始からブランチ作成、コミット、PR作成、マージ、タスク完了までの一連の開発フローを自動化・支援し、開発者が本質的なコード作成に集中できる環境を実現する。既存のタスク管理ツール(Trello、Asana、Jira)のような重厚さは排除し、開発者が本当に必要な機能だけを提供する。

### 1.4 目的

1. **コンテキストスイッチの削減**: ターミナル↔GUIの切り替えをゼロにし、集中力を維持
2. **開発フローの自動化**: タスク管理とGit操作を一体化し、手作業を削減
3. **チーム連携の効率化**: CLIでチーム全体の状況を即座に確認可能に
4. **タスク管理の時間削減**: タスク管理にかかる時間を50%削減

## 2. 解決する課題

### 2.1 現状の問題点

#### 問題1: 画面切り替えによる集中力の低下
- **現状**: ターミナルでコード作成 → ブラウザでTrello/Asana → ターミナルに戻る
- **影響**: 1日20回以上の切り替えで集中力が途切れる
- **測定**: 画面切り替え回数(1日平均20回)、集中時間の中断回数

#### 問題2: タスクとコードの乖離
- **現状**: どのタスクのためにどのブランチで作業しているか不明確
- **影響**: コミットメッセージへのタスク番号記載漏れ、PR作成時の手作業
- **測定**: タスク番号記載漏れ率(現状30%)、PR作成の手動ステップ数

#### 問題3: 手作業が多い
- **現状**: タスク完了 → 手動でステータス更新 → 手動でPR作成 → 手動でタスク完了
- **影響**: 本質的でない作業に時間を浪費
- **測定**: タスク管理に費やす時間(現状1日30分)

#### 問題4: チーム状況の可視性不足
- **現状**: チームメンバーの作業状況を知るにはGUIツールを開く必要がある
- **影響**: ペアプロやレビュー時に即座の状況確認ができない
- **測定**: チーム状況確認の頻度と所要時間

## 5. 機能要件

### 5.1 P0(MVP - 必須機能)

#### 5.1.1 タスクの基本操作

**ユーザーストーリー**:
個人開発者として、ターミナルから離れずにタスクを管理するために、基本的なCRUD操作ができる必要がある。

**コマンド仕様**:
```bash
# タスク作成
task add "ユーザー認証機能の実装"
task add "バグ修正: ログイン画面" --priority high --due 2025-01-20

# タスク一覧
task list
task list --status in_progress
task list --priority high

# タスク詳細
task show 1

# タスク完了
task done 1

# タスク削除
task delete 1
```

**受け入れ条件**:
- [ ] `task add`で新規タスクを作成できる(タイトルは必須)
- [ ] タイトルのみの最小限のコマンドで作成可能(`task add "title"`)
- [ ] オプションで優先度(`--priority`)、期限(`--due`)を指定可能
- [ ] `task list`でタスク一覧を表形式で表示(ID、ステータス、タイトル、ブランチ)
- [ ] `task show <id>`で詳細情報を表示(作成日時、更新日時、優先度、期限、説明)
- [ ] `task done <id>`でタスクを完了状態に変更
- [ ] `task delete <id>`で確認プロンプト後にタスクを削除
- [ ] コマンド実行時間100ms以内(平均的なPC環境)

**優先度**: P0(必須)

#### 5.1.2 タスクとGitブランチの自動連携

**ユーザーストーリー**:
個人開発者として、タスク作業時にブランチを自動作成・切り替えしたいため、`task start`でブランチ連携を自動化する必要がある。

**コマンド仕様**:
```bash
# タスク開始(ブランチ自動作成・切り替え)
task start 1
# → feature/task-1-user-authentication ブランチを作成して切り替え

# 現在作業中のタスクを表示
task current

# タスク終了(ブランチはそのまま)
task stop
```

**ブランチ命名規則**:
- フォーマット: `feature/task-{id}-{slug}`
- 例: `feature/task-1-user-authentication`
- slug: タイトルから自動生成(小文字、ハイフン区切り、最大50文字)

**受け入れ条件**:
- [ ] `task start <id>`でタスクのステータスが`in_progress`に変更
- [ ] ブランチが存在しない場合は自動作成(命名規則に従う)
- [ ] ブランチが既に存在する場合は切り替えのみ
- [ ] `task current`で現在作業中のタスク情報を表示
- [ ] `task stop`でタスクのステータスを`open`に戻す(ブランチは維持)
- [ ] Gitリポジトリがない場合はエラーメッセージを表示
- [ ] 未コミットの変更がある場合は警告を表示

**優先度**: P0(必須)

#### 5.1.3 タスクのステータス管理

**ユーザーストーリー**:
個人開発者として、タスクの進捗状況を把握するために、明確なステータス管理が必要である。

**ステータス定義**:
- `open`: 新規作成、未着手
- `in_progress`: 作業中(`task start`で自動設定)
- `completed`: 完了(`task done`で設定)
- `archived`: アーカイブ済み(一覧に表示されない)

**コマンド仕様**:
```bash
# ステータス変更
task status 1 in_progress
task status 1 completed

# アーカイブ
task archive 1

# アーカイブ済みタスクを含めて表示
task list --all
```

**受け入れ条件**:
- [ ] タスク作成時のデフォルトステータスは`open`
- [ ] `task start`で自動的に`in_progress`に変更
- [ ] `task done`で自動的に`completed`に変更
- [ ] `task archive`でステータスを`archived`に変更
- [ ] `task list`はデフォルトで`archived`を除外
- [ ] `task list --all`で全ステータスを表示
- [ ] 無効なステータスを指定した場合はエラーメッセージを表示

**優先度**: P0(必須)

#### 5.1.4 シンプルなタスク一覧表示

**ユーザーストーリー**:
個人開発者として、タスクの全体像を即座に把握するために、見やすい一覧表示が必要である。

**表示フォーマット**:
```
ID  Status       Title                      Branch
1   in_progress  ユーザー認証機能の実装      feature/task-1-user-authentication
2   open         データエクスポート機能      -
3   completed    初期セットアップ           feature/task-3-initial-setup
```

**フィルタリング・ソート**:
```bash
# ステータスでフィルタ
task list --status in_progress
task list --status open,in_progress

# 優先度でフィルタ
task list --priority high

# 優先度でソート
task list --sort priority

# 作成日時でソート
task list --sort created
```

**受け入れ条件**:
- [ ] 表形式で見やすく表示(ID、ステータス、タイトル、ブランチ)
- [ ] ステータスごとに色分け(in_progress: 黄、completed: 緑、open: 白)
- [ ] `--status`オプションで特定ステータスのみ表示
- [ ] `--priority`オプションで特定優先度のみ表示
- [ ] `--sort`オプションでソート順を指定可能
- [ ] タスクが0件の場合は分かりやすいメッセージを表示
- [ ] 1000件のタスクでも1秒以内に表示

**優先度**: P0(必須)

#### 5.1.5 データ永続化

**ユーザーストーリー**:
個人開発者として、作成したタスクが失われないように、ローカルにデータを保存する必要がある。

**データ保存形式**:
- フォーマット: JSON
- 保存場所: `.task/tasks.json`
- Gitで管理可能(チーム共有可能)

**データ構造例**:
```json
{
  "tasks": [
    {
      "id": 1,
      "title": "ユーザー認証機能の実装",
      "status": "in_progress",
      "priority": "medium",
      "due": "2025-01-20",
      "branch": "feature/task-1-user-authentication",
      "created_at": "2025-01-10T10:00:00Z",
      "updated_at": "2025-01-10T11:30:00Z"
    }
  ]
}
```

**受け入れ条件**:
- [ ] 初回実行時に`.task/`ディレクトリを自動作成
- [ ] タスク作成・更新・削除時に自動保存
- [ ] JSONファイルが破損している場合はエラーメッセージを表示
- [ ] `.task/`ディレクトリが削除された場合は自動再作成
- [ ] データ保存失敗時は元のデータを保持(ロールバック)

**優先度**: P0(必須)

### 5.2 P1(重要 - 初期リリース後すぐに追加)

#### 5.2.1 GitHub Issuesとの連携

**ユーザーストーリー**:
個人開発者として、GitHub Issuesで管理しているタスクをローカルと同期したいため、双方向連携機能が必要である。

**コマンド仕様**:
```bash
# GitHub Issuesからインポート
task import --github

# ローカルタスクをGitHub Issueとして作成
task export --github 1

# 双方向同期
task sync
```

**受け入れ条件**:
- [ ] GitHub Personal Access Tokenで認証
- [ ] `task import --github`でGitHub Issuesを取得してローカルに保存
- [ ] `task export --github <id>`でローカルタスクをGitHub Issueとして作成
- [ ] `task sync`で双方向同期(ローカル変更をGitHub へ反映、GitHub変更をローカルへ反映)
- [ ] GitHub APIエラー時は分かりやすいエラーメッセージを表示

**優先度**: P1(重要)

#### 5.2.2 タスク完了時の自動処理

**ユーザーストーリー**:
個人開発者として、タスク完了時の定型作業を自動化したいため、ワンコマンドで完了処理を実行できる必要がある。

**コマンド仕様**:
```bash
task done 1 --auto
# 以下を自動実行:
# 1. 現在のブランチをmainにマージ
# 2. リモートにプッシュ
# 3. (オプション) GitHub PRを自動作成
# 4. タスクステータスをcompletedに変更
```

**受け入れ条件**:
- [ ] `--auto`オプションでマージ、プッシュ、PR作成を自動実行
- [ ] 危険な操作(マージ)前に確認プロンプトを表示
- [ ] マージコンフリクトが発生した場合はエラーを表示して中断
- [ ] GitHub PR作成は設定で有効化した場合のみ
- [ ] 各ステップの実行結果を表示

**優先度**: P1(重要)

#### 5.2.3 タスクの絞り込み・検索

**ユーザーストーリー**:
個人開発者として、特定のタスクを素早く見つけるために、検索機能が必要である。

**コマンド仕様**:
```bash
# キーワード検索
task search "認証"

# 担当者でフィルタ(将来的なチーム機能用)
task list --assignee me

# 複合条件
task list --status in_progress --priority high
```

**受け入れ条件**:
- [ ] `task search <keyword>`でタイトルと説明を部分一致検索
- [ ] 検索結果を一覧形式で表示
- [ ] 複数のフィルタ条件を組み合わせ可能
- [ ] 検索結果が0件の場合は分かりやすいメッセージを表示

**優先度**: P1(重要)

#### 5.2.4 優先度と期限の管理

**ユーザーストーリー**:
個人開発者として、緊急度の高いタスクを優先的に処理するために、優先度と期限を管理したい。

**優先度レベル**:
- `low`: 低
- `medium`: 中(デフォルト)
- `high`: 高
- `critical`: 緊急

**コマンド仕様**:
```bash
# 優先度と期限を指定してタスク作成
task add "緊急バグ修正" --priority critical --due 2025-01-20

# 優先度順にソート
task list --sort priority

# 期限が近い順にソート
task list --sort due

# 今日が期限のタスクのみ表示
task list --due today

# 期限切れのタスクを表示
task list --overdue
```

**受け入れ条件**:
- [ ] タスク作成時に優先度と期限を指定可能
- [ ] `--sort priority`で優先度順にソート(critical > high > medium > low)
- [ ] `--sort due`で期限が近い順にソート
- [ ] `--due today`で今日が期限のタスクのみ表示
- [ ] `--overdue`で期限切れタスクを表示(期限切れは赤色で表示)
- [ ] 期限が設定されていないタスクは最後に表示

**優先度**: P1(重要)

### 5.3 P2(できれば - 将来的に検討)

#### 5.3.1 チーム機能

**ユーザーストーリー**:
チームリーダーとして、チームメンバー全員のタスク状況を把握するために、チーム向け機能が必要である。

**コマンド仕様**:
```bash
# チーム全体のタスク一覧
task list --team

# タスクをメンバーにアサイン
task assign 1 @alice

# メンバーごとのタスク一覧
task list --assignee @alice
```

**受け入れ条件**:
- [ ] `.task/config.json`でチームメンバーを設定
- [ ] `task assign`でタスクをメンバーにアサイン
- [ ] `task list --team`でチーム全体のタスク一覧を表示
- [ ] メンバーごとに色分けして表示

**優先度**: P2(できれば)

#### 5.3.2 作業時間の記録

**ユーザーストーリー**:
個人開発者として、タスクにかかった時間を記録したいため、作業時間計測機能が必要である。

**コマンド仕様**:
```bash
# タスク開始(時間計測開始)
task start 1

# タスク完了(作業時間を記録)
task done 1
# 出力: タスク完了! 作業時間: 2時間30分

# 作業時間の確認
task show 1
# 出力: 合計作業時間: 5時間15分
```

**受け入れ条件**:
- [ ] `task start`で作業時間の計測を開始
- [ ] `task stop`で計測を停止(タスクは継続)
- [ ] `task done`で最終的な作業時間を記録
- [ ] `task show`で合計作業時間を表示
- [ ] 複数回の作業セッションに対応(累積時間を記録)

**優先度**: P2(できれば)

#### 5.3.3 タスクのテンプレート

**ユーザーストーリー**:
個人開発者として、定型的なタスク(バグ修正、機能追加など)を素早く作成するために、テンプレート機能が必要である。

**コマンド仕様**:
```bash
# テンプレート作成
task template create bug-fix

# テンプレートを使用してタスク作成
task add --template bug-fix "ログイン画面のバグ修正"

# テンプレート一覧
task template list
```

**受け入れ条件**:
- [ ] `task template create <name>`でテンプレートを作成
- [ ] テンプレートには優先度、期限、説明のデフォルト値を設定可能
- [ ] `task add --template <name>`でテンプレートを使用してタスク作成
- [ ] テンプレート一覧を表示可能

**優先度**: P2(できれば)

## 6. 非機能要件

### 6.1 パフォーマンス

#### 応答時間
- **コマンド実行時間**: 100ms以内(平均的なPC環境、タスク数1000件以下)
- **タスク一覧表示**: 1000件でも1秒以内
- **検索**: キーワード検索が500ms以内(タスク数1000件)

#### スケーラビリティ
- **最大タスク数**: 10,000件まで快適に動作
- **大規模リポジトリ**: Git操作が3秒以内に完了

**測定方法**:
- ベンチマークテストで各操作の実行時間を計測
- 1000件、5000件、10000件のタスクでパフォーマンステスト

### 6.2 ユーザビリティ

#### 学習コストの低さ
- **初回利用**: 新規ユーザーが5分以内に基本操作(add/list/start/done)を習得
- **ヘルプの充実**: `task --help`で全コマンドの説明を確認可能
- **エラーメッセージ**: 分かりやすく、修正方法を提示

#### 操作の直感性
- **コマンド体系**: Gitコマンドに類似した命名規則
- **最小限のキー入力**: エイリアス対応(`task a`は`task add`のエイリアス)
- **タブ補完**: Bash/Zshでタブ補完をサポート

**測定方法**:
- ユーザビリティテスト(5人の新規ユーザーで計測)
- ヘルプコマンドの完全性チェック

### 6.3 信頼性

#### データ保護
- **データ損失ゼロ**: 操作前に自動バックアップ(`.task/backups/`)
- **ロールバック**: エラー発生時は元の状態に復元
- **データ検証**: JSON破損時は最新のバックアップから復元

#### エラーハンドリング
- **Gitエラー**: Git操作失敗時は元の状態に戻す
- **ファイルシステムエラー**: ディスク容量不足、権限エラーを検出
- **ネットワークエラー**: GitHub API失敗時は明確なエラーメッセージ

**測定方法**:
- エラーケースのテストカバレッジ80%以上
- データ損失テスト(異常終了、ディスク容量不足など)

### 6.4 セキュリティ

#### 認証情報の保護
- **GitHub Token**: 環境変数または安全なキーチェーンに保存
- **平文保存禁止**: 認証情報をJSONファイルに平文で保存しない

#### 危険な操作の確認
- **削除操作**: `task delete`は確認プロンプトを表示
- **マージ操作**: `task done --auto`はマージ前に確認
- **強制実行**: `--force`オプションで確認をスキップ可能

**測定方法**:
- セキュリティチェックリストで検証
- 認証情報の保存方法を監査

## 7. 技術的制約と前提条件

### 7.1 前提条件

#### ユーザー環境
- **Gitインストール済み**: Git 2.30以上がインストールされている
- **Node.js**: Node.js 18以上がインストールされている
- **Gitリポジトリ**: 作業ディレクトリがGitリポジトリである

#### Gitリポジトリがない環境への対応
- `task`コマンドはGitリポジトリがない場合でも基本操作(add/list/done)は動作
- Git連携機能(`task start`など)はエラーメッセージを表示

### 7.2 技術スタック

#### 開発言語・フレームワーク
- **言語**: TypeScript 5.x
- **CLIフレームワーク**: Commander.js(シンプルで学習コストが低い)
- **Git操作**: simple-git(Node.js用Gitライブラリ)
- **GitHub API**: Octokit(GitHub公式ライブラリ)

#### データ保存
- **フォーマット**: JSON(`.task/tasks.json`)
- **理由**: シンプル、Gitで管理可能、特別なソフトウェア不要
- **将来的な移行**: 必要に応じてSQLiteに移行可能

### 7.3 制約事項

#### パフォーマンス制約
- **タスク数上限**: 10,000件(それ以上は動作保証外)
- **Git操作**: 大規模リポジトリ(1GB以上)では動作が遅くなる可能性

#### 機能制約
- **GitHub専用**: 初期バージョンはGitHubのみ対応(GitLab、Bitbucketは将来対応)
- **単一リポジトリ**: 複数リポジトリを横断したタスク管理は未対応

## 9. リスクと対策

### 9.1 技術的リスク

#### リスク1: 大規模リポジトリでのパフォーマンス低下
- **影響度**: 中
- **発生確率**: 中
- **対策**: ベンチマークテストで事前検証、パフォーマンス改善のバッファを設ける
- **緩和策**: 警告メッセージを表示、キャッシュ機構の導入

#### リスク2: Git操作の失敗(マージコンフリクトなど)
- **影響度**: 高
- **発生確率**: 高
- **対策**: エラーハンドリングの徹底、ロールバック機能の実装
- **緩和策**: 危険な操作前に確認プロンプトを表示

#### リスク3: データ破損・損失
- **影響度**: 高
- **発生確率**: 低
- **対策**: 自動バックアップ、データ検証の実装
- **緩和策**: バックアップからの復元手順をドキュメント化

### 9.2 プロダクトリスク

#### リスク1: ユーザーの習得コストが高い
- **影響度**: 高
- **発生確率**: 中
- **対策**: ユーザビリティテストで検証、チュートリアルの充実
- **緩和策**: `task init`でインタラクティブなセットアップ

#### リスク2: 既存ツール(Linear、Jira)からの移行が難しい
- **影響度**: 中
- **発生確率**: 中
- **対策**: インポート機能の提供、移行ガイドの作成
- **緩和策**: 既存ツールと並行利用できる設計

#### リスク3: チーム機能の需要が不明確
- **影響度**: 中
- **発生確率**: 中
- **対策**: MVPは個人向けに絞る、ユーザーフィードバックで需要を検証
- **緩和策**: P2として将来的に追加

### 9.3 ビジネスリスク

#### リスク1: 競合ツールとの差別化が不十分
- **影響度**: 高
- **発生確率**: 中
- **対策**: Git統合の深さで差別化、ターミナル完結の徹底
- **緩和策**: ユーザーインタビューで競合優位性を検証

#### リスク2: ユーザー獲得が進まない
- **影響度**: 高
- **発生確率**: 中
- **対策**: 開発者コミュニティでの宣伝、GitHub Trendingを狙う
- **緩和策**: 早期フィードバックループを確立

## 10. 用語集

| 用語 | 定義 |
|------|------|
